# Release Skill

整理项目图片、更新 README 时间轴、提交并推送到远程。

## 项目结构规范

```
MyStoryWithCC/
├── avatar.jpg            # 头像，保持不动
├── YYYY/
│   └── MM/              # 例如 2026/03/
│       ├── YYYYMMDDxx.jpg       # 单图发布，例如 2026031001.jpg
│       ├── YYYYMMDDxx/          # 多图发布子目录，例如 2026031202/
│       │   ├── YYYYMMDDxx-1.jpg #   第1张，例如 2026031202-1.jpg
│       │   └── YYYYMMDDxx-2.jpg #   第2张，例如 2026031202-2.jpg
│       └── YYYYMM.md            # 当月对话记录，例如 202603.md
└── README.md
```

**图片命名规则**：
- **单图发布**：`YYYYMMDD序号.jpg`，直接放在 `YYYY/MM/` 下
  - `2026031001.jpg` = 2026年3月10日第1次发布（单图）
- **多图发布**：`YYYYMMDD序号/YYYYMMDD序号-子序号.jpg`，放在以序号命名的子目录下
  - `2026031202/2026031202-1.jpg` = 2026年3月12日第2次发布的第1张
  - `2026031202/2026031202-2.jpg` = 2026年3月12日第2次发布的第2张

**对话记录文件格式**（`YYYYMM.md`）：

```markdown
# YYYY年M月

---

## YYYY-MM-DD

### 01

#### **豪哥：**

用户的问题内容

#### **CC：**

CC 的回复内容

**回复内小节标题**（用加粗，不用 # 标题）

正文...

---

### 02

#### **豪哥：**
...
```

层级规范：
- `#` 年月标题
- `##` 日期
- `###` 对话编号（01、02……）
- `#### **豪哥：**` / `#### **CC：**` 说话人
- 回复内部分段用 `**加粗文字**`，不用 `####` 标题

## 执行步骤

### 第一步：扫描并移动错位图片

扫描整个项目目录，找出所有 `.jpg` / `.jpeg` / `.png` 文件（排除 `avatar.jpg` 和 `.git/` 目录）。

判断文件是否在正确位置：

**单图**（`YYYYMMDD序号.扩展名`，无 `-子序号`）：
- 必须位于 `YYYY/MM/` 目录下（例如 `2026031001.jpg` 应在 `2026/03/`）

**多图**（`YYYYMMDD序号-子序号.扩展名`，含 `-数字` 后缀）：
- 必须位于 `YYYY/MM/YYYYMMDD序号/` 子目录下（例如 `2026031202-1.jpg` 应在 `2026/03/2026031202/`）
- 若根目录或 `YYYY/MM/` 下出现 `-子序号` 文件，视为错位

对于每个错位的图片：
1. 解析文件名前8位得到年份（前4位）和月份（第5-6位）
2. 单图：创建 `YYYY/MM/` 并移动；多图：创建 `YYYY/MM/YYYYMMDD序号/` 并移动
3. 告知用户移动了哪些文件

文件名不符合规范的图片（无法从前8位解析出合法日期的）：提示用户手动处理，不要强制移动。

### 第二步：更新 README.md 时间轴

扫描所有 `YYYY/MM/` 目录下的图片，重新生成 README.md 的时间轴部分。

**时间轴格式**：

```markdown
## 🗓️ 时间轴

### YYYY 年

#### YYYYMM · 月份汉字

| 图片 | 日期 | 合集&标签 |
|------|------|----------|
| ![文件名不含扩展名](YYYY/MM/文件名.jpg) | YYYY-MM-DD | 合集名 · #标签1 #标签2 |
```

**多图发布的 README 规则**：同一次发布（子目录）只在时间轴中展示第1张图（`-1`），路径指向子目录内：
```markdown
| ![2026031202-1](2026/03/2026031202/2026031202-1.jpg) | 2026-03-12 | |
```

**月份汉字对照**：一月、二月、三月、四月、五月、六月、七月、八月、九月、十月、十一月、十二月

**排序规则**（倒序，最新在前）：
- 年份从晚到早
- 月份从晚到早
- 同一月内按文件名倒序（即日期+序号从大到小，`2026031202-2` > `2026031202-1` > `2026031202` > `2026031201`）

**合集列表**（目前只有一个，后续用户会告知新增）：
- AI

**合集&标签处理规则**：
- 更新时间轴时，**保留**现有行的"合集&标签"内容，不得清空或覆盖
- 新增的图片行，"合集&标签"列留空，等待用户补充
- 合集名与标签之间用 ` · ` 分隔，多个标签用空格分隔（如 `AI · #大模型 #英伟达`）

更新时，只替换 README.md 中 `## 🗓️ 时间轴` 开头直到文件结尾的部分。文件头部（标题、项目介绍、目录结构说明）保持不变。

### 第三步：更新对话记录（YYYYMM.md）

找到本月的对话记录文件（例如 `2026/03/202603.md`），将本次会话中**尚未被记录**的对话追加进去。

**判断哪些对话需要记录**：
- 对比现有文件内容，找出本次会话中新发生的对话轮次
- 若今天的日期（`## YYYY-MM-DD`）在文件中已存在，将新对话追加到该日期章节的**末尾**（即该日期下已有编号之后，下一个 `##` 日期之前）
- 若今天的日期不存在，在文件末尾新增日期章节后再追加

**格式严格遵循规范**（参见上方"对话记录文件格式"）：
- `###` 编号从 `01` 开始，同一天内续接已有最大编号
- `#### **豪哥：**` 写用户原话（可适当精简，保留核心意思）
- `#### **CC：**` 写 CC 的回复（保留完整内容，内部小节用 `**加粗**` 而非 `####`）

如果本次会话没有值得记录的实质对话（只有操作指令），可跳过此步骤。

### 第四步：Git 提交

**暂存文件**：
```bash
git add .
```

**检查 git status**，生成 commit message：

Commit message 格式遵循 Conventional Commits：
- 只移动了图片：`chore: 整理图片到正确目录`
- 只更新了 README：`docs: 更新时间轴`
- 只更新了对话记录：`docs: 更新对话记录`
- 图片 + README + 对话记录都有：`chore: 整理图片并更新时间轴`
- 有对话记录更新时，用 body 补充日期和对话数，例如：
  ```
  docs: 更新对话记录

  新增 2026-03-12 共 2 段对话
  ```

如果 git status 是干净的（没有变化），告知用户"项目已是最新状态，无需提交"，跳过提交和推送。

**提交**：
```bash
git commit -m "..."
```

### 第四步：推送到远程

```bash
git push
```

推送成功后，告知用户本次 release 完成，并简要总结：移动了几张图片、更新了哪些月份的时间轴、commit hash 前几位。
